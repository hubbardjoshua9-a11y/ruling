# Reference — Assessment Task Types (the classification at STAGE-2)

Classify every task as one primary type. The type sets the **default resistance** and tells you which failure patterns to look for. A task can carry a secondary type (e.g. an AI-integrated record task) — rule on the primary, note the secondary.

| ID | Type | Default resistance | The bar it must clear | Watch for |
|---|---|---|---|---|
| **T1** | **Knowledge question** (short answer, written) | LOW — generatable by default | Anchored to context AI can't hold, *or* paired with a verification prompt | Generic phrasing (RULE-10); self-undermining "or general knowledge" (RULE-09); prohibition with no enforcement (RULE-14) |
| **T2** | **Template / portfolio completion** | LOW–MEDIUM | Either contextualised to the student's **own** data + verification, or treated as integrated with judgment fields | Scenario *provided* → student just populates (RULE-08); no prompt/annotation fields if AI-integrated (RULE-24) |
| **T3** | **Applied project** (planning or performance) | MEDIUM | Coherence + a verification anchor on the resist parts | Looks resistant because it's long; still generatable from a provided scenario |
| **T4** | **Written roleplay** (typed scenario responses) | LOW — text is what AI produces | Cannot clear it as written | **Structural mismatch** → REBUILD to recorded/observed (RULE-12) |
| **T5** | **Recorded / observed performance** (video, live) | HIGH — hard to outsource | A *responsive* element (unscripted question/challenge) + the right person present | Scripted video (student reads a script) → REVISE, add responsiveness (RULE-17) |
| **T6** | **Oral questioning / professional conversation** | HIGH | Unscripted, probing, live or async-on-unseen-prompt | A fixed published question list students can pre-generate answers to |
| **T7** | **AI-integrated** (use AI, then judge it) | Depends entirely on design | Requires AI use, captures the prompt, and assesses the student's **judgment over** the output | AI output *is* the submission with no evaluation (RULE-22); AI "optional" (RULE-23); no prompt field (RULE-24) |
| **T8** | **Record / data task** (logs, registers, aggregate data) | LOW–MEDIUM | Draws on data the student genuinely produced/holds; a reflective or override element | Records filled from a provided scenario; aggregate report sitting on top of generatable inputs |
| **T9** | **Roleplay simulation** (AI persona or assessor-in-role) | HIGH if a human grades a live response | Human grades it (never the AI); live student response; conditions allow a simulated environment | Bot as *sole* evidence where conditions need a real person → may be FLAG/REVISE |

---

## How the type drives the ruling
- **T4 is the only type that is structurally wrong on its face.** Typed roleplay can't produce the evidence the unit wants (interaction, real-time judgment) — that's a **REBUILD**, not a patch, because no instruction added to a typed task makes it a performance.
- **T5/T6/T9** are the workhorses of resistance — but only with the *responsive* element. A recorded monologue read from a script (T5 done wrong) is closer to T1 than to a performance.
- **T7** has the widest spread: a well-built AI-integrated task is COMPLIANT; the same task with the AI output as the submission and no judgment requirement is REVISE.
- **T1/T2/T8** are generatable by default and usually need either own-data contextualisation *or* an attached verification point to pass.

## Performance vs knowledge evidence (the line that triggers REBUILD)
A type can be *valid for knowledge* and *invalid for performance*. If a task gathers **knowledge evidence** (a written explanation) where the unit requires **performance evidence** (a demonstration), no amount of REVISE fixes it — the evidence type is wrong → **REBUILD** (RULE-20). Classifying the type correctly is what surfaces this. See `reference/evidence-requirements.md`.
